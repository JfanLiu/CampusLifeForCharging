## Background / Problem

现有项目是 React Native 原生应用，核心功能依赖移动端运行时，包括本地存储、原生导航、相机扫码和支付宝 SDK。用户希望将其部署为服务器上的 Web 服务，使浏览器可直接访问。

## Scope

- 新增 `web/` 目录，提供可独立部署的 Web 前后端一体版本
- 复用现有上游协议、加密方式和 XML/GBK 解码逻辑
- 提供登录、会话保持、余额查询、充电状态查询、手动充电、浏览器扫码、充值入口、设置页
- 提供所有地点的充电状态总览，并尽可能展示每一根充电桩的状态
- 由服务器代理上游接口，避免浏览器直接暴露协议细节

## Non-goals

- 不重构现有 React Native 客户端
- 不替换上游业务接口
- 不引入数据库持久化；会话默认使用进程内存
- 不承诺在桌面浏览器中获得与原生支付宝 SDK 完全一致的支付体验

## Constraints

- 上游接口使用 DES-CBC + PKCS7 加密，并返回 XML 包裹的 GBK 文本
- 登录态原实现基于本地 `userid` 存储，Web 版需改为服务端会话
- 支付接口当前仅观察到 `paytype: 1` 的 App 支付链路，Web 无法直接复用原生 SDK
- 原版充值流程经反编译确认是 `payapply -> orderinfo -> Alipay SDK payV2(...)`
- 逐桩状态需要先通过 `getlist` 拿地点，再通过隐藏接口 `getsublist(rid)` 逐个地点拉取
- 当前仓库无现成 Web 构建链，需新增最小可部署结构

## High-level Plan

1. 在 `web/server` 中实现上游协议封装、会话管理和 HTTP API
2. 在 `web/public` 中实现无需构建的静态单页应用
3. 使用浏览器相机能力替代 React Native 扫码能力
4. 为充值流程提供 Web 入口，并对上游仅支持 App 支付的限制做准确提示
5. 增加所有地点状态总览与逐桩状态视图
6. 补充部署与运行说明，确保可在独立服务器上启动

## Verification

- 能在浏览器中登录并保持会话
- 能查询余额和充电状态
- 能手动输入二维码后发起充电
- 能通过浏览器相机识别二维码并回填
- 能从 Web 页面发起充值流程，并给出可观察的支付结果或限制说明
- 能查看所有地点的当前状态，并看到每一根充电桩当前是否空闲、充电中或故障

## Rollback

- Web 版本为新增目录，不影响现有 React Native 工程
- 回滚时可直接删除 `web/` 与 `docs/web-service/`，以及根目录中新增的 Web 说明
